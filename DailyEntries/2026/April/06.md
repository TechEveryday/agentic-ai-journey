## Lessons Discovered

### 1. Node Version Incompatibility (Critical)
- **Problem**: Tests failed with "crypto.getRandomValues is not a function"
- **Root Cause**: Node 16.15.1 too old; dependencies require Node 18+
- **Solution**: Added .nvmrc pinning Node to 20.11.0, ran `nvm use`
- **Lesson**: Always pin Node version explicitly; don't rely on system default

### 2. Test Isolation & Scoping (React Testing Library)
- **Problem**: Component tests found wrong elements (multiple elements with same selector)
- **Root Cause**: Global DOM queries instead of container-scoped queries
- **Solution**: Changed from `screen.getByPlaceholderText()` to `container.querySelector()`
- **Lesson**: Always scope queries to the component under test, not the entire DOM

### 3. Async State Updates in Tests (React)
- **Problem**: "act() warning" - state updates not wrapped properly
- **Root Cause**: Async operations in handlers not wrapped in act()
- **Solution**: Used `act()` wrapper and `waitFor()` for async assertions
- **Lesson**: React requires explicit act() wrapping for async state changes in tests

### 4. Git Workflow & Feature Branches (Team Discipline)
- **Problem**: Initial work committed to main branch (violates best practices)
- **Root Cause**: No enforced feature branch workflow
- **Solution**: Created feature/todo-checklist branch, all work isolated from origin/main
- **Lesson**: Feature branches required for all work; main is release branch only

### 5. React 19 & Node 20+ Standards (Project Consistency)
- **Problem**: Future projects might use older React/Node versions
- **Root Cause**: No explicit version requirements documented
- **Solution**: Created node-react-versions.md lesson, updated all prompts
- **Lesson**: Always use React 19+ and Node 20+ for new projects

## Reflection

### What Went Well

1. **Plan-Driven Implementation** — Detailed 7-step plan before coding eliminated false starts and kept focus sharp
2. **Incremental Testing Strategy** — Tests written alongside each layer (core → infrastructure → application → presentation) validated design early
3. **Clean Architecture Discipline** — 4-layer separation enforced via TypeScript path aliases prevented coupling and made Phase 2 design trivial
4. **Comprehensive Documentation** — Architecture docs, future phases with code examples, and Docker setup all written same day
5. **Interface-Based Design** — ITodoRepository abstraction meant local/HTTP swap for Phase 2 requires changing exactly one file
6. **Material-UI Component Library** — Saved implementation time with professional components (edit-in-place, loading states, error handling)

### What Didn't Go Well

1. **Initial Node Version Oversight** — Started without .nvmrc pinning, caused test failures (Node 16 vs 18+ mismatch). Resolved by adding .nvmrc and updating SESSION_START.md
2. **Test Isolation Issues** — First few component tests had DOM scoping problems (global queries). Learned React Testing Library best practice of container-scoped selectors
3. **Async State in Tests** — act() warnings on first integration test run. Solved by wrapping async operations and using waitFor() properly
4. **Git Workflow Not Enforced** — Initial work on main (before lesson documented feature branches). Now enforced in prompts and SESSION_START.md

### Technical Insights

- **Composition Over Inheritance** — React hooks + interface-based repos provides better flexibility than class inheritance for swapping implementations
- **Async Interface Design** — Making all repository methods async (even though localStorage is sync) enables Phase 2 HTTP swap without changing signatures
- **Validation as Boundary** — Putting validation logic in core layer makes it reusable in C# backend without duplication
- **Path Aliases for Architecture** — TypeScript `@/core`, `@/infrastructure`, etc. path aliases make layered imports explicit and prevent circular dependencies
- **Material-UI Theming** — Default MUI theme + CSS override patterns sufficient for production UI without custom styling burden
