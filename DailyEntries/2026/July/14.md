## Overview

The README listed nine projects to tackle, each a one-line title with no requirements. Rather than build nine shallow MVPs, we planned all nine and built three: **Contacts**, **Tournament/Bracket Maker**, and **2-way Chat** — the last being the repo's first real backend, an ASP.NET Core 10 SignalR service.

All three were built in parallel by separate Sonnet agents, each in its own git worktree on its own branch. Every project passes lint, build, unit, integration, and E2E gates, verified independently rather than taken on the agents' word. **226 tests total, all passing.**

The more durable output may be the two lessons this session produced. Both came from things that only surface when you actually *run* the tooling: the reference project's `lint` and `test:e2e` scripts had never worked, and running three agents in parallel exposed a shared-port defect that let one E2E suite run against a completely unrelated application.

## Context and Planning

Nine one-line titles is not a specification. Before any code, we resolved the ambiguities via a planning pass, then got the plan approved.

### Key Decisions

- **Plan nine, build three** - `lessons/ai-as-solo-generator.md` documents the cost of letting AI run unsupervised at scale. Building three first validates the template before it's replicated nine times. The other six have committed plans in `planning/`.
- **React 19, not 18** - `lessons/node-react-versions.md` marks React 19+ as CRITICAL, but the reference (`todo-checklist`) ships React 18.3.1. The lesson wins; the reference is now flagged as needing a fix.
- **MUI stays at v5** - Verified via `npm view`: `@mui/material@5.18.0` declares `react: "^17 || ^18 || ^19"` as a peer. React 19 + MUI 5 is clean. An earlier assumption that this would require MUI v6/v7 was simply wrong.
- **.NET 10, not 9** - The plan approved .NET 9 on the belief it was current. Checking Microsoft's support policy showed .NET 9 (STS) ends Nov 10 2026 — four months out — while .NET 10 (LTS) runs to Nov 2028. Since this backend becomes the template for the future Service Creator project, it was switched to 10.
- **In-memory storage for chat, not Postgres** - `IMessageRepository` is the seam. Postgres adds EF Core, migrations, a container, and integration plumbing to what was already the largest item. Deferred deliberately, not forgotten.
- **Two of nine remain blocked, and say so** - Social Media App has no differentiator from Instagram Clone + 2-way Chat. Roblox Game has no Lua runtime installed and no game concept. Writing plans that pretend otherwise would be worse than writing none.

## Work Completed

### Contacts (`projects/contacts/`, branch `feature/contacts`)

CRUD contacts with search. The closest analogue to `todo-checklist`, chosen deliberately as the project that proves the React 19 template.

- `core/`: `Contact`, `createContact`, `validateContact`, `matchesQuery` — all pure.
- `application/`: `IContactRepository` (all async), `useContacts` hook.
- `infrastructure/`: `LocalStorageContactRepository`, key `contacts:contacts`.
- `presentation/`: `ContactsPage` (composition root), plus form/list/item/search components.
- Search filtering is derived in the page via the pure `matchesQuery` predicate — deliberately *not* in the repository, so it's unit-testable without storage.

### Tournament/Bracket Maker (`projects/tournament-bracket/`, branch `feature/tournament-bracket`)

Single-elimination brackets. The domain logic is the point; the UI is the thin part.

- Recursive **standard seeding order** so seed 1 and seed 2 can only meet in the final (`order(2n)` expands each seed `s` into `[s, 2n+1−s]`).
- Byes for non-power-of-2 fields, landing on top seeds as a consequence of the seeding order — asserted in tests, not assumed.
- `advanceWinner` is pure and returns a new bracket; input immutability is a test.
- Documented ambiguity decisions: 0 and 1 participants both rejected; invalid advancement throws rather than silently returning unchanged.

### 2-way Chat (`projects/two-way-chat/`, branch `feature/two-way-chat`)

The repo's **first backend**, and the largest item in the batch.

- **Backend** (`net10.0`, Clean Architecture): `Chat.Domain` (zero-dependency `Message`, `MessageValidator`) → `Chat.Application` (`ChatService`, `IMessageRepository`) → `Chat.Infrastructure` (thread-safe `InMemoryMessageRepository`) + `Chat.Api` (`ChatHub`).
- Server stamps `Id` and `SentAt` — the client is never trusted for them.
- CORS locked to the frontend origin with `AllowCredentials()`, which SignalR requires.
- **Frontend**: `IMessageTransport` port in `application/`, `SignalRMessageTransport` in `infrastructure/`. The `useChat` hook never imports `@microsoft/signalr` — it depends only on the port, which is what makes it unit-testable against a mock.

- `planning/` — six self-contained plans for the deferred projects.
- `.gitignore` at root — the repo had none; agent worktrees were showing up as untracked.

## Technical Implementation

### The port that makes a backend swappable

Every repository interface is async even when the implementation is synchronous. This looks like overhead until the day it isn't:

```ts
export interface IContactRepository {
  getAll(): Promise<Contact[]>;      // localStorage is sync — this is deliberate
  getById(id: string): Promise<Contact | null>;
  save(contact: Contact): Promise<void>;
  delete(id: string): Promise<void>;
}
```

The same discipline carried into C#, which is what will let chat swap in Postgres later without touching the hub or the UI:

```csharp
public interface IMessageRepository {
    Task<IReadOnlyList<Message>> GetByRoomAsync(string roomId);
    Task AddAsync(Message message);
}
```

### Seeding, expressed as a pure recursion

```ts
// order(2)  = [1, 2]
// order(4)  = [1, 4, 2, 3]
// order(8)  = [1, 8, 4, 5, 2, 7, 3, 6]
// Each seed s in order(n) expands to the pair [s, 2n+1-s].
// Consequence: byes land on top seeds for free — no special-casing.
```

This is the kind of logic worth having an agent write *and* worth testing hard: it's easy to write something plausible that's subtly unfair.

## Lessons Discovered

Two new lessons, both documented in `lessons/`:

- **[parallel-agents-and-shared-ports](../../../lessons/parallel-agents-and-shared-ports.md)** - Git worktrees isolate files. They do not isolate ports. All three projects inherited `baseURL: 5173` + `reuseExistingServer: true`, so Playwright would happily reuse *whatever* was answering on that port. The bracket agent's first E2E run pointed at an unrelated "Soccer Roster" app that wasn't even from this repo. It failed loudly that time — but two similar apps could produce a run that passes against the wrong one. **When you parallelize, re-examine every assumption the serial version was allowed to make.**

- **[template-drift-and-broken-scripts](../../../lessons/template-drift-and-broken-scripts.md)** - The reference project's `npm run lint` and `npm run test:e2e` had **never worked**: no `eslint.config.js` exists (ESLint 9 requires flat config), and `playwright.config.ts` sits where the root-run script can't find it, with a `testDir` that double-resolves. Both scripts looked authoritative when read and failed instantly when run. `npm test` was green because it's the only gate CLAUDE.md enforces — and it excludes E2E. **Run a reference's scripts before cloning it N times.**

A third, smaller finding: all three agents independently discovered RTL's auto-cleanup wasn't firing reliably under Vitest globals, and each added an explicit `afterEach(cleanup)`. Three agents converging on the same fix is a template gap, not a coincidence.

## Reflection

### What Went Well

1. **Planning before building caught four wrong assumptions** - React/MUI compatibility, the .NET version, two undefined project titles, and a missing Lua runtime. Each would have been discovered mid-build at much higher cost.
2. **Worktree isolation worked exactly as intended** - Three agents, three branches, zero file conflicts, three clean single commits.
3. **The agents self-corrected on real bugs** - MUI v6-only `slotProps` used against v5; `aria-label` on a MUI `TextField` landing on the wrapper div instead of the input; E2E clearing `localStorage` before navigation and throwing `SecurityError`. All found and fixed during their own verification, and all reported rather than hidden.
4. **Verifying independently was worth it** - Not because the agents lied (they didn't — the numbers held up), but because the port defect meant a green E2E from a parallel run wasn't *evidence* of anything until re-run serially.

### What Didn't Go Well

1. **The .NET SDK install failed on sudo** - `brew install --cask dotnet-sdk` shells out to the system installer and needs a password; a non-interactive session can't supply one. Resolved by the user running it manually. Worth knowing before planning any future toolchain install.
2. **Assumed a support date instead of checking it** - .NET 9 was asserted to be EOL in May 2026. It isn't; it's supported to Nov 10 2026. The conclusion (use .NET 10) survived, but the reasoning was wrong until verified. Version and support facts should be looked up, never recalled.
3. **The port collision was a self-inflicted harness bug** - Not in any agent's code. In the decision to run three port-identical projects concurrently.

### Technical Insights

- **The dependency rule is worth enforcing across languages.** `presentation → application → core` in TypeScript and `Api → Application → Domain` in C# are the same rule. Chat proves it holds on both sides of the wire.
- **A port is the cheapest possible seam.** `IMessageTransport` meant the chat hook could be fully unit-tested against a mock with no server, while `SignalRMessageTransport` — the one file that can't be meaningfully unit-tested — is isolated and covered by E2E instead.
- **`npm test` passing is not the same as the project working.** In this repo it's `vitest run`, which explicitly excludes `tests/e2e/**`. The CLAUDE.md gate is necessary, not sufficient.
- **macOS holds port 5000.** Control Center's AirPlay Receiver binds it. Chat's backend was configured for 5000 and worked here only because the bind addresses differed. Fragile; see Next Session Notes.
